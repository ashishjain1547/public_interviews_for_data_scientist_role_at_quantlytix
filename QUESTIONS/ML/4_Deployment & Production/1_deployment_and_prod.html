How would you deploy a model for real-time inference? Locally and in the cloud?

Deploying a machine learning model for real-time inference involves setting up an architecture where your model can respond to prediction requests instantly (low-latency). Here's how it's done in two main environments: locally and in the cloud.

‚úÖ 1. Deploying Locally for Real-Time Inference

üß∞ Stack:

  • Model: Trained .pkl, .h5, or .joblib file

  • Framework: Flask / FastAPI

  • Serving: Local server with REST API

üß™ Steps:

  1. Save the model

joblib.dump(model, 'model.pkl')

  1. Create a REST API using FastAPI

import joblib
import pandas as pd

app = FastAPI()
model = joblib.load('model.pkl')

@app.post("/predict")

    df = pd.DataFrame([data])
    pred = model.predict(df)
    return {"prediction": pred.tolist()}

  1. Run the API

bash
uvicorn app:app --reload

  1. Test with curl or Postman

bash
curl -X POST http://localhost:8000/predict -H "Content-Type: application/json" -d '{"feature1": 0.5, "feature2": 1.2}'

What steps would you take if your deployed model‚Äôs accuracy starts dropping?

When you notice your deployed model‚Äôs accuracy (or other key performance metrics) degrading in production, it‚Äôs usually a symptom of one (or more) of these issues: data drift, concept drift, data quality problems, or pipeline/configuration errors. Here‚Äôs a systematic playbook to diagnose and remediate:

1. Alerting & Monitoring

  1. Automatic Alerts
    ‚Ä¢ Monitor live metrics (accuracy, precision/recall, response time) via dashboards (e.g. Grafana, CloudWatch).
    ‚Ä¢ Set up alerts for significant drops (e.g. > 5 % below baseline).

  2. Logging
    ‚Ä¢ Capture input distributions, model outputs, and ground-truth labels (when available).
    ‚Ä¢ Log upstream data changes (feature distributions, schema changes).

2. Root-Cause Analysis

  1. Check Data Drift
    ‚Ä¢ Compare current feature distributions against training data (e.g. using population-stability index or KS test).
    ‚Ä¢ Look for shifts in means, variances, or new/unseen categories.

  2. Check Concept Drift
    ‚Ä¢ Verify whether the underlying relationship between features and targets has changed.
    ‚Ä¢ Retrain a lightweight ‚Äúshadow‚Äù model periodically and compare performance.

  3. Data Quality & Pipeline
    ‚Ä¢ Confirm no corruption in the feature-engineering pipeline (e.g. missing one-hot columns, misaligned timestamps).
    ‚Ä¢ Validate that upstream systems aren‚Äôt introducing nulls or odd outliers.

  4. Configuration & Versioning
    ‚Ä¢ Ensure the correct model version is deployed.
    ‚Ä¢ Confirm inference code, dependencies, and environment match those you tested.

3. Immediate Mitigations

  1. Rollback
    ‚Ä¢ If a recent deployment coincides with the drop, roll back to the last stable model/version.

  2. Fallback Model
    ‚Ä¢ Serve a simpler, more robust model (e.g. logistic regression or previous checkpoint) until the issue is resolved.

4. Long-Term Remediation

  1. Retraining Pipeline
    ‚Ä¢ Automate scheduled retraining (e.g. nightly/weekly) using the latest labeled data.
    ‚Ä¢ Include validation with a hold-out set to catch regressions before promotion.

  2. Online Learning / Continual Learning
    ‚Ä¢ If appropriate, implement incremental updates so the model adapts as new data arrives.

  3. Enhanced Drift Detection
    ‚Ä¢ Integrate libraries like River or Alibi Detect for real-time drift monitoring and alerts.

  4. Feature-Store Versioning
    ‚Ä¢ Use a feature store (e.g. Feast) to enforce consistent feature computation between training and serving.

  5. A/B Testing & Canary Releases
    ‚Ä¢ Deploy new model versions to a small percentage of traffic.
    ‚Ä¢ Compare performance against the incumbent before full rollout.

5. Feedback Loop & Learning

  1. Label Feedback
    ‚Ä¢ Build mechanisms for collecting true labels (e.g. periodic human review, user feedback) and feeding them back to the pipeline.

  2. Post-Mortem
    ‚Ä¢ Document the root cause, mitigation steps, and improvements made.
    ‚Ä¢ Update runbooks so future teams can react faster.

  3. Continuous Improvement
    ‚Ä¢ Regularly revisit your feature set, model architecture, and hyperparameters as business needs evolve.

In summary, a drop in production accuracy isn‚Äôt just a ‚Äúretrain the model‚Äù signal‚Äîit‚Äôs a prompt to investigate drift, data quality, deployment integrity, and monitoring gaps. Building robust pipelines, automated drift detection, and clear rollback paths will help you recover quickly and prevent recurrence.

How do you track and compare model performance over time?

Tracking and comparing model performance over time is essential for maintaining a healthy ML system in production. Here‚Äôs how you can do it systematically:

‚úÖ 1. Define Key Performance Indicators (KPIs)

Track relevant metrics aligned with your business and ML objectives:

  • Classification: Accuracy, Precision, Recall, F1-score, ROC-AUC

  • Regression: MSE, RMSE, MAE, R¬≤

  • Ranking: NDCG, MAP, Precision@K

  • Drift: Feature distribution shift, prediction shift, label shift

Use multiple metrics to avoid false confidence in a single number.

‚úÖ 2. Set Up Experiment Tracking

Use an experiment tracking tool to log training and evaluation runs:

Tools:

  • MLflow

  • Weights & Biases

  • Neptune

  • Comet

  • TensorBoard (for deep learning models)

Track:

  • Model version & hash

  • Hyperparameters

  • Training data version

  • Evaluation metrics

  • Artifacts (model file, plots)

‚úÖ 3. Create a Performance Dashboard

Set up a dashboard (using tools like Grafana, Streamlit, or Power BI) to:

  • Visualize model performance over time

  • Compare across versions (V1, V2, V3‚Ä¶)

  • Detect anomalies or drops in metrics

  • Monitor real-time inference latency and throughput

‚úÖ 4. Track Data and Model Drift

Use statistical tests or drift-detection libraries:

  • Alibi Detect

  • River

  • Evidently AI

  • scikit-multiflow

Track:

  • Feature distribution changes (e.g., PSI, KL divergence)

  • Target drift if labels are available

  • Concept drift (relation between X and y changes)

‚úÖ 5. Version Everything

Model Versioning

  • Store models using DVC, MLflow, S3, GCS, etc.

  • Tag models with a version ID, git commit, date

Data Versioning

  • Use tools like DVC, LakeFS, or snapshots in your data warehouse

  • Ensure full reproducibility of past experiments

‚úÖ 6. Monitor in Production

Set up monitoring tools to track:

  • Live prediction distributions

  • Latency and throughput

  • Error rates

  • Feedback loop data (true labels when available)

Tools like Prometheus + Grafana, Seldon Core, Triton Inference Server, or custom logging can help.
